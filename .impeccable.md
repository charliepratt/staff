# TypeyTypey Design Context

## Design Context

### Users
Writers of all kinds — screenwriters, novelists, playwrights, bloggers. Ranging from aspiring to working professionals. They want a tool that respects their craft without demanding technical expertise. They are not power users who enjoy configuring software; they want to open it and write. They may be on a 13" laptop on a plane or a wide desktop monitor at home. Their context is creative focus — they need the tool to disappear so the writing can happen.

### Brand Personality
**Warm, focused, approachable.**

The interface should feel like a calm writing studio — no-nonsense, distraction-free, with subtle nods that writers will love (proper formatting, typewriter fonts, page-feel). It should never feel like enterprise software, a developer tool, or a SaaS dashboard.

### Emotional Goals
- **Calm** — the writer feels at ease, not overwhelmed by options
- **Confident** — formatting is handled correctly; the writer trusts the tool
- **Delighted (subtly)** — small touches that make a writer smile without being showy

### Aesthetic Direction

**Visual tone**: Warm minimalism. Clean but not cold. Paper-inspired surfaces, ink-inspired accents. Every pixel should feel considered but never precious.

**References**:
- **Vellum** — the gold standard for aesthetic writing tools. Live preview, beautiful typography, the feeling of writing on a real page. The closest to the target feel.
- **Ulysses** — appreciated for its clean interface, but TypeyTypey must avoid its Markdown-first approach. Writers should never see markup.
- **iA Writer** — focus mode, typography-first thinking. Good energy, but too minimal for TypeyTypey's multi-format ambitions.

**Anti-references**:
- **Final Draft** — dated, cluttered, noisy. The UI feels like it hasn't evolved in a decade. Feature-rich but visually exhausting.
- **Scrivener** — powerful but overwhelming. Too many panels, too many options visible at once. The opposite of calm.
- **Generic SaaS/dashboard UIs** — cool grays, sharp borders, developer-tool energy. This is a writing app, not a data product.

**Theme**: Light mode is primary. Dark mode is on the roadmap and should be planned for (design tokens already support swapping). All color choices should work in both contexts.

### Design Tokens (Current)

```
Surfaces:      #FDFCFA (paper), #F5F3EF (linen), #E8E6E1 (canvas)
Text:          #2C2825 (primary), #6B6560 (secondary), #9C9590 (tertiary)
Borders:       #E5E0D8 (subtle), #DDD8D0 (default)
Accent:        #3D5A80 (ink blue), #2E4A6E (hover)
States:        #B54848 (error), #4A7C59 (success)
UI Font:       DM Sans (400, 500, 700)
Editor Font:   Courier Prime (default), 9 alternatives across mono/sans/serif
Icons:         Lucide React, regular weight, 14-16px
```

### Active States
Subtle warm background tints (bg-border-1 / #E5E0D8), not heavy color reversals. Active elements use dark text on a warm background, never white-on-blue. The active state should feel like a gentle press, not a highlight reel.

### Design Principles

1. **The writing surface is sacred.** The editor area gets maximum space, minimum chrome. Every UI element must justify its presence against the alternative of more writing space.

2. **Warm over cool, always.** Cream over gray. Charcoal over black. Ink blue over electric blue. The palette should feel like paper, leather, and wood — not steel, glass, and neon.

3. **Subtle delight over obvious flourish.** A gentle highlight flash when jumping to a scene. A soft shadow on the page break line. The kind of details a writer notices on the third use and appreciates — never the kind that impress on first glance and annoy on the hundredth.

4. **Format fidelity is non-negotiable.** What the writer sees must match what gets printed/exported. Page proportions, margins, typography — these aren't approximate, they're exact. Writers trust tools that get formatting right.

5. **No Markdown, ever.** The writer types words, not syntax. Every formatting decision is made through direct manipulation, keyboard shortcuts, or smart detection — never through markup characters in the text.

### Accessibility
- Target: **WCAG AA** compliance
- All interactive elements must have visible focus states
- Color choices must meet AA contrast ratios (4.5:1 for normal text, 3:1 for large text)
- Keyboard navigation must work for all features
- Reduced motion preferences should be respected
- Minimum body text: 14px (already set in design system)

### Component Patterns
- **Buttons**: Radix-based, three variants (default, ghost, accent). Warm active states.
- **Dropdowns/Menus**: Radix DropdownMenu and ContextMenu. Warm surfaces, no heavy borders.
- **Tooltips**: Radix Tooltip. Dark background (text-1), light text (surface-1). Brief, helpful.
- **Panels**: Slide transitions (150ms ease-out). No jarring show/hide.
- **Modals/Drawers**: Slide from edge, subtle backdrop. Smooth enter/exit transitions.
